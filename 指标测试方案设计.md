### 1.2 关键约束识别

- **约束 1**：3 台虚拟机共 12 Slots，node02/03 仅 2 核 3.7GB → DS2 并行度调整空间有限（1~8），ML Tuner 搜索空间窄。
- **约束 2**：Flink + RocketMQ 混部 → Savepoint/Restart 期间 Broker I/O 被抢占，影响延迟测量。
- **约束 3**：Host-Only 虚拟网络 → "网络传输延迟降低 20%" 不能直接用物理网络指标，建议改为"Flink Source 到 Sink 的处理路径延迟降低"。
- **约束 4**：模拟数据高度同质 → 需人为构造多阶段变速率场景才能体现弹性调度价值。

### 1.3 指标口径统一定义（实验前必须锁定）

| 指标名称 | 精确定义 | 测量点 | 计算方式 |
|----------|---------|-------|---------|
| **端到端时延** | Producer 发送时刻 → Sink 写出时刻 | Producer 端在 Protobuf 中写入 `send_ts`；Sink 端记录 `sink_ts` | `P99(sink_ts - send_ts)` |
| **DS2 调度响应时间** | 检测到需要调整 → 新并行度 Job 恢复处理第一条数据 | ds2_controller 日志时间戳 | `t_first_output - t_decision` |
| **DS2 调度精度** | 推导并行度 vs 理论最优并行度 | 稳态吞吐 / 单并行度容量 | `|p_actual - p_optimal| / p_optimal × 100%` |
| **缓存命中率提升** | W-TinyLFU 命中率 vs FIFO 命中率 | FeatureExtraction 日志 | `(hit_wtlfu - hit_fifo) / hit_fifo × 100%` |
| **传输延迟降低** | 缓存命中时跳过重复计算的延迟节省 | 算子处理延迟 Gauge | `(latency_nocache - latency_cache) / latency_nocache × 100%` |
| **ML Tuner 效果** | 优化后 J 值（加权综合得分）vs 默认参数 J 值 | observation_store 记录 | `(J_default - J_best) / J_default × 100%` |

### 阶段 0：实验基础设施准备（预计 1 天）

#### 0.1 端到端时延埋点改造

**问题**：当前 Producer 写入 `SeismicRecord.timestamp` 是原始数据秒级时间戳，不是发送时刻，无法用于端到端时延计算。

**改造步骤**：

**步骤 1**：在 `seismic_record.proto` 中增加字段：

```protobuf
int64 producer_send_ts_ms = 20;  // Producer 发送时毫秒时间戳
```

**步骤 2**：`SeismicProducer.java` 在发送前填充：

```java
record.setProducerSendTsMs(System.currentTimeMillis());
```

**步骤 3**：在 `EventFeatureSink` / `GridSummarySink` 中计算并暴露 Gauge：

```java
long e2eLatency = System.currentTimeMillis() - record.getProducerSendTsMs();
// 维护滑动窗口计算 P50 / P99，通过 Flink Gauge 暴露
// 指标名: seismic_e2e_latency_p50 / seismic_e2e_latency_p99
```

**步骤 4**：确保 3 台虚拟机 NTP 时钟同步，偏差 < 10ms：

```bash
sudo ntpdate pool.ntp.org && sudo hwclock -w
```

#### 0.2 DS2 调度时间戳埋点

**步骤 1**：`ds2_controller.py` 在每次触发调度决策入口处记录：

```python
t_decision = time.time()
```

**步骤 2**：Savepoint + Cancel + Resubmit 完成后，轮询 Flink Metrics API，当目标算子 `numRecordsOut > 0` 时记录：

```python
t_recovery = time.time()
response_time = t_recovery - t_decision
```

**步骤 3**：将每次调度事件写入 `ds2_experiment_log.csv`，字段如下：

```
t_decision, p_before, p_recommended, p_applied, t_recovery, response_time, throughput_before, throughput_after, backlog_before, backlog_after
```

#### 0.3 统一实验参数基线

所有实验共用以下默认配置，对比实验仅改变单一变量：

#### 1.4 结果分析要点

### 阶段 2：DS2 弹性调度实验（预计 2~3 天）

#### 2.1 实验目标

#### 2.2 实验 E2-0：单并行度容量标定（前置实验）

**目的**：获取理论最优并行度的计算基准 `C`（单并行度容量），用于后续精度误差计算。

**方法**：固定 `signal-extraction` 算子并行度为 p，逐步提高输入速率直到吞吐饱和。

**操作流程**：

```
对 p ∈ {1, 2, 4, 8}，分别执行：
  Step1: 提交 Job，固定目标算子并行度为 p
  Step2: 以初始速率启动 Producer，稳定 60s 后记录吞吐
  Step3: 逐步提高速率（每档提高 500 rec/s，持续 30s）
  Step4: 当 numRecordsOut 不再随输入增加时，记录饱和吞吐 T_sat(p)
```

**标定结果记录表**：

```
| 并行度 p | 饱和吞吐 T_sat(p) (rec/s) | 备注        |
|---------|------------------------|------------|
| 1       |                        | 单并行度容量 C |
| 2       |                        |            |
| 4       |                        |            |
| 8       |                        |            |
```

**理论最优并行度计算公式**：

```
p_optimal = ceil( R_input / C × headroom_ratio )
其中 headroom_ratio = 1.8（来自 ds2_config.yaml）
```

#### 2.5 实验 E2-3：调度精度计算

对 E2-1 每个调度决策点，使用 E2-0 标定的 `C` 计算精度误差：

```
p_optimal(i) = ceil( R_input(i) / C × 1.8 )
p_ds2(i)     = DS2 推导并行度
error(i)     = |p_ds2(i) - p_optimal(i)| / p_optimal(i) × 100%

汇总统计：
  平均误差 = mean(error)
  最大误差 = max(error)
  满足 ≤15% 的决策占比
```

#### 2.7 潜在风险与应对策略

**风险 1**：Savepoint + Restart 链路本身耗时 > 10s。

```
应对：
  a. 预先测量空载 Savepoint 耗时，若 > 5s 需减小状态后端存储量
  b. 考虑将 state_backend 改为 hashmap 减少序列化开销
  c. 若确实超标，在论文中如实说明并分析瓶颈原因（虚拟机环境限制）
```

**风险 2**：3-VM 环境下 DS2 调整空间太窄（1~8），调度档位太少。

```
应对：
  a. 聚焦 2→4→8 三档验证，已足够说明弹性调度能力
  b. 在论文中注明：生产环境下集群规模更大，可线性扩展至更细粒度调整
```

**风险 3**：冷却期内负载再次变化触发抖动。

```
应对：
  a. 确保阶梯负载每个阶段持续时间 > cooldown_s（60s）
  b. 即每个负载档位持续至少 90s（60s 冷却 + 30s 稳态采集）
```

#### 3.2 传输延迟降低的口径定义

由于实验环境为 Host-Only 虚拟网络，无法测量真实物理传输延迟。

**重新定义口径**：

```
"传输延迟降低" = FeatureExtractionFunction 算子处理延迟降低比

物理含义：
  缓存命中 → 跳过节点历史特征的重复计算 → 算子处理耗时缩短
  → 直接体现为 Flink 端到端延迟减少
  → 等价于减少了"无效计算导致的传输阻塞时间"
```

#### 3.3 实验 E3-1：无缓存 vs W-TinyLFU 处理延迟对比

**改造方案**：在 `FeatureExtractionFunction.processElement()` 中增加处理时间埋点：

```java
long startNs = System.nanoTime();
// ... 原有处理逻辑 ...
long costNs = System.nanoTime() - startNs;
// 通过 Flink Gauge 暴露：feature_processing_latency_ns
```

**实验分组**：

```
组 A（NoCache 基线）：
  --cache-strategy NONE
  FeatureExtractionFunction 禁用缓存，每条数据走完整计算路径
  运行：2000 节点 × 正常速率 × 180s，取稳态后 120s 数据

组 B（W-TinyLFU）：
  --cache-strategy WTLFU
  运行：相同配置
```

**采集指标**：

```
feature_processing_latency_ns 的 P50 / P99（每 10s 采样）
seismic_e2e_latency_p99（端到端延迟对比）
```

**结果记录表**：

```
| 指标                      | 组 A (NoCache) | 组 B (W-TinyLFU) | 降低比 (%) |
|--------------------------|--------------|-----------------|----------|
| P50 算子处理延迟 (μs)      |              |                 |          |
| P99 算子处理延迟 (μs)      |              |                 |          |
| 端到端 P99 延迟 (ms)       |              |                 |          |
| 稳态缓存命中率 (%)          | N/A          |                 | N/A      |
```

**延迟降低比计算**：

```
processing_latency_reduction = (latency_A - latency_B) / latency_A × 100%
e2e_latency_reduction        = (e2e_A - e2e_B) / e2e_A × 100%
```

### 阶段 4：ML Tuner 实验（预计 2~3 天）

#### 4.1 实验目标

验证预期指标 ④：作业参数能随地震数据特征实现较优的自适应调整。

#### 4.2 实验前置检查

在正式运行 ML Tuner 前，需确认以下条件：

```
✅ verify_step1_metrics.py 通过：所有算子指标端点可访问
✅ verify_step2_values.py 通过：关键指标数值正常（非 NaN / 0）
✅ verify_step3_savepoint.py 通过：Savepoint 触发链路正常
✅ verify_step4_restart.py 通过：完整重启链路端到端正常
✅ DS2 Controller 已关闭（避免与 ML Tuner 的 Job Restart 冲突）
```

#### 4.3 参数空间确认

ML Tuner 搜索的 3 维参数空间：

```
| 参数名                  | 范围              | 默认值    | 含义           |
|-----------------------|-----------------|---------|--------------|
| checkpoint_interval_ms | [10000, 120000] | 30000   | Checkpoint 间隔|

| watermark_idle_ms      | [500, 10000]    | 3000    | 水位线空闲超时   |
```

综合得分 J 的 4 个子目标：

```
J = w1×(吞吐得分) + w2×(延迟得分) + w3×(CPU得分) + w4×(CKP耗时得分)
各权重来自 ml_tuner_config.yaml
```

#### 4.6 潜在风险与应对策略

**风险 1**：每轮迭代耗时过长，30 轮无法在合理时间内完成。

```
应对：
  a. 预先测量单轮 Savepoint + Restart + 稳态等待的耗时
  b. 若单轮 > 5 分钟，将 max_rounds 降至 20 轮
  c. 将稳态等待时间从 60s 压缩至 45s（在论文中注明）
```

**风险 2**：GP 代理模型在观测点较少时方差过大，EI 推荐质量差。

```
应对：
  a. 冷启动阶段确保至少 5 轮 LHS 覆盖参数空间边角
  b. 若 J 值无收敛趋势，切换为 RF 代理模型（surrogate_model.py 已支持）
  c. 在论文中说明：受虚拟机资源限制，搜索空间和观测点数量有限，
     实际生产环境中更多迭代次数可带来更优结果
```

**风险 3**：ML Tuner 推荐参数效果不优于默认参数。

```
应对：
  a. 分析各子目标分值，找出 ML Tuner 在哪个子目标上有改善
  b. 即使 J 值提升有限，收敛曲线本身已能说明 BO 的探索-利用平衡能力
  c. 在论文中侧重论述"自适应调整"能力，而非绝对提升幅度
```

## 五、特别注意事项

### 5.1 虚拟机环境的局限性说明（论文中需声明）

```
1. 宿主机资源共享：3 台虚拟机运行在同一物理机上，
   CPU 调度存在抢占，实测延迟可能高于生产环境。

2. 存算混部：Flink TM 与 RocketMQ Broker 共节点，
   Checkpoint I/O 与消息持久化存在磁盘争用。

3. 网络虚拟化：Host-Only 网络延迟特征与真实物理网络不同，
   "传输延迟"指标以算子处理延迟为替代口径。

4. 数据规模：2000 节点仿真数据总量约为生产场景的 1/5 至 1/10，
   DS2 并行度上限受 12 Slots 约束。

→ 上述局限性不影响方法论的有效性，
  各模块的相对改善比例在生产环境中具有参考价值。
```

### 5.2 实验可重复性保障

```
每轮实验前执行以下重置操作：
  a. 停止所有 Flink Job
  b. 清空 RocketMQ Topic 积压：
     mqadmin deleteTopic -n localhost:9876 -t seismic-raw
     重新创建 Topic
  c. 清空 Flink 状态目录：
     rm -rf /tmp/flink-checkpoints/*
  d. 等待 30s 让系统资源释放
  e. 重新启动 Flink 集群
```

### 5.3 指标未达标时的应急预案

```
① 端到端延迟 > 5s：
   → 检查 RocketMQ 消费积压，调小 buffer_timeout_ms 至 50ms
   → 检查 Checkpoint 是否产生 I/O 毛刺，延长间隔至 60s

② DS2 响应时间 > 10s：
   → 分析 Savepoint 耗时占比，若 > 7s 则在论文中单独分析瓶颈
   → 说明：在生产环境 SSD + 独立部署场景下，Savepoint 耗时可压缩至 2~3s

③ 缓存命中率提升 < 20%：
   → 复查 parse_log.py 的日志提取逻辑，确认统计口径一致
   → 已有实验数据显示提升约 26%，此风险较低

④ ML Tuner J 值无正向提升：
   → 分析各子目标分值，找出有改善的维度单独说明
   → 侧重 BO 收敛曲线的探索-利用过程分析，而非绝对提升幅度
```

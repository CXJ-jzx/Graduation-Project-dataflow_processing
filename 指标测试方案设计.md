## 四、Phase 0：测量基础设施搭建

### 4.1 端到端延迟测量修正

#### 4.1.1 问题描述

当前 `EventFeatureSink` 中 `latency_p99` 的计算基准为 `.seis` 文件中原始时间戳（`timeStamp` 字段），该时间戳为数据生成时的历史时间。Producer 循环读取文件时，该时间戳与当前系统时间差值极大，导致延迟计算无实际意义。

#### 4.1.2 修正方案

**修正位置**：`SeismicProducer.java`

**修正逻辑**：在构建 Protobuf `SeismicRecord` 时，使用当前系统时间覆盖 `timeStamp` 字段。

```text
修正前: record.setTimestamp(从文件解析的 timeStamp)
修正后: record.setTimestamp(System.currentTimeMillis() / 1000)
```

> 保持秒级精度，与 Sink 端 `record.getTimestamp() * 1000` 的计算逻辑一致。

#### 4.1.3 延迟分段埋点（可选，用于论文精细分析）

| 分段 | 埋点位置 | 计算方式 |
|------|----------|----------|
| T_produce→source | Source 算子内部 | `System.currentTimeMillis() - record.getTimestamp() * 1000` |
| T_source→filter | HardwareFilter 输出 | 同上 |
| T_filter→signal | SignalExtraction 输出 | 同上 |
| T_signal→feature | FeatureExtraction 输出 | 同上（即现有 latency_p99） |
| T_feature→sink_ack | Sink 回调确认 | 回调时间 - record.getTimestamp() * 1000 |

### 4.2 指标可用性验证

使用已有的 `verify_step1_metrics.py` 脚本，确认以下指标在 Flink REST API 中可正常获取：

| 指标名 | 所属算子 | 用途 |
|--------|----------|------|
| `numRecordsInPerSecond` | 各算子 | 吞吐量 |
| `numRecordsOutPerSecond` | 各算子 | 吞吐量 |
| `busyTimeMsPerSecond` | 各算子 | DS2 核心输入 |
| `l2_hit_rate` | FeatureExtraction | 缓存命中率 |
| `l2_occupancy` | FeatureExtraction | 缓存空间利用率 |
| `l3_hit_rate` | GridAggregation | 缓存命中率 |
| `l3_occupancy` | GridAggregation | 缓存空间利用率 |
| `latency_p99` | EventFeatureSink | 端到端延迟 |

**验证标准**：所有指标返回非空且数值合理。

### 4.3 统一指标采集脚本

编写或复用 Python 指标采集脚本，定时（每 10 秒）从 Flink REST API 拉取上述全部指标，写入 CSV 文件，格式如下：

```csv

```

### 5.3 实验 1.3：数据完整性验证

#### 目的

验证 Exactly-Once 语义下数据不丢不重。

#### 方法

1. Producer 端统计发送总条数 `N_send`
2. HardwareFilter SideOutput 端统计过滤条数 `N_filter`
3. SignalExtraction 无信号段过滤条数 `N_no_signal`（如果有）
4. EventFeatureSink 统计输出总条数 `N_output`

#### 验证公式

```text
N_send = N_output + N_filter + N_no_signal + N_lost
数据完整性 = N_lost == 0
```

#### 输出

- 各环节数据量统计表
- 数据完整性结论

### 6.1 实验矩阵

| 实验编号 | 缓存策略 | 负载水平 |

### 6.2 每组实验采集指标

| 指标 | 说明 |
|------|------|
| L2 缓存命中率 | `l2_hit_rate`，取稳定后均值 |

### 6.3 指标计算公式

```text

延迟降低 = (Lat_baseline - Lat_FALRU) / Lat_baseline × 100%
其中 Lat_baseline = 同一负载下其他策略的平均端到端延迟
```

### 6.4 达标条件

- 缓存命中率提升 ≥ 20%
- 延迟降低 ≥ 20%

## 七、Phase 3：DS2 弹性调度实验（指标 2）

### 7.1 实验 3.1：调度响应时间测量

#### 目的

测量从负载突变到 DS2 完成扩缩容的总耗时。

#### 实验步骤

```text
1. 启动 Flink 作业（默认并行度 Source=2, 算子=4, Sink=2）
2. 启动 DS2 Controller
3. Producer 以 L1（~200 条/s）运行 5 分钟
4. 突然切换 Producer 到 L3（~2000 条/s）
5. 从 DS2 日志中记录：
   - T1: DS2 首次检测到需要扩容的时刻
   - T2: Savepoint 完成时刻
   - T3: 新作业恢复 RUNNING 的时刻
   - 调度响应时间 = T3 - T1
6. 待系统稳定后，切换回 L1，记录缩容响应时间
7. 重复 3 次取均值
```

#### 达标条件

- 扩容响应时间 ≤ 10s
- 缩容响应时间 ≤ 10s

#### 输出

- 每次扩缩容的 T1 / T2 / T3 时间表
- 平均响应时间

### 7.2 实验 3.2：调度精度测量

#### 7.2.1 基准建立（手动穷举）

对每个负载水平，手动逐一测试不同并行度，记录吞吐和延迟：

| 负载水平 | 测试并行度范围 | 每个配置运行时长 |
|----------|---------------|-----------------|
| L1（~200 条/s） | p = 1, 2, 3, 4 | 5 分钟 |
| L2（~760 条/s） | p = 1, 2, 3, 4, 5, 6 | 5 分钟 |
| L3（~2000 条/s） | p = 2, 3, 4, 5, 6, 7, 8 | 5 分钟 |

**P_optimal 定义**：在延迟 ≤ 5s 约束下，吞吐量达标的最小并行度。

#### 7.2.2 精度计算

```text
对每个负载水平:
  精度误差 = |P_ds2 - P_optimal| / P_optimal × 100%

总体精度误差 = 各负载水平精度误差的均值
```

#### 达标条件

- 总体精度误差 ≤ 15%

#### 输出

- 各负载水平：P_optimal vs P_ds2 对比表
- 精度误差计算表

### 7.3 实验 3.3：动态负载波动测试

#### 目的

验证 DS2 在持续波动负载下的稳定性，观察是否出现调度振荡。

#### 负载变化时间线

```text
时间轴（分钟）:
0───5───10───15───20───25───30
 L1   L3    L2    L3    L1   L2
(低) (高)  (中)  (高)  (低) (中)
```

#### 采集指标

| 指标 | 说明 |
|------|------|
| DS2 每次决策的目标并行度 | 从 DS2 CSV 日志提取 |
| 实际执行的扩缩容次数 | 统计 scaling 日志 |
| 每次扩缩容后的稳定延迟 | Flink Metrics |
| 是否出现振荡 | 连续 3 次以上交替扩缩 |

#### 输出

- 并行度随时间变化的阶梯图（X: 时间，Y: 并行度）
- 叠加负载水平变化的背景色
- 延迟随时间变化曲线

### 7.4 实验 3.4：与阈值调度策略的对比

#### 基线策略实现

简单阈值调度规则：

```text
IF avg_cpu > 80%  THEN p = min(p + 1, max_p)
IF avg_cpu < 30%  THEN p = max(p - 1, min_p)
检查周期: 15s（与 DS2 一致）
```

在实验 3.3 相同的动态负载模式下，分别运行阈值调度和 DS2 调度，对比：

| 对比维度 | 阈值调度 | DS2 |
|----------|----------|-----|
| 总扩缩容次数 | | |
| 平均响应时间 | | |
| 平均调度精度误差 | | |
| 稳定后平均延迟 | | |
| 是否出现振荡 | | |

#### 输出

- 两种策略的并行度变化对比图
- 对比维度汇总表
